## 2）分层逻辑架构

API 层
- code/api.yml：OpenAPI 接口契约定义。
- code/kg_rag.py：HTTP 接口处理、入参校验、状态码映射。

运行时层
- code/kg_rag_runtime.py：KGRAGService 懒加载单例、Neo4j 初始化异常格式化。

核心服务层
- code/kg_rag_core/service/core.py：KGRAGService 组合入口。
- code/kg_rag_core/service/pipeline.py：问答主流程编排。
- code/kg_rag_core/service/det_*.py：规则化（确定性）问答分支。
- code/kg_rag_core/service/ops.py：通用基础能力（CSV、提示词、推理调用、索引检查等）。

持久化与检索层
- code/kg_rag_core/repository.py：Neo4j 图能力与向量检索仓储封装。
- code/graph_building.py：图构建与向量嵌入生成。

配置与映射层
- code/kg_rag_core/settings.py：环境变量与提示词模板。
- code/synonyms.py：列名同义词与问题同义词归一化。

数据与评测层
- data/*.csv：建图与回退校验的数据真值来源。
- test00/*.py：批量回归与自动评测脚本。

## 3）主运行流程

A. 建图流程
1. POST /create-fmea-graph -> code/kg_rag.py:create_graph
2. 运行时单例 -> get_rag_service()
3. KGRAGService.create_fmea_graph() -> graph_building.create_fmea_graph()
4. CSV 归一化 -> 在 Neo4j 中 MERGE 节点与关系
5. create_vector_embeddings() -> 生成 Chunk 向量并建立索引关联

B. 问答流程
1. POST /question-answer -> code/kg_rag.py:answer_question
2. 进入 pipeline.py 中的 KGRAGService.answer_question()
3. 前置检查：问题有效性、Neo4j 可用性、图是否已初始化
4. 优先走 deterministic 分支（高精度查表类问题）
5. 若未命中 deterministic：
  - 让 LLM 生成 Cypher 并执行图查询
  - 若为空或无效，回退向量相似检索
6. 对检索上下文做摘要压缩
7. 生成最终问答结果
8. 将答案保存到 answer/ 下带时间戳的文件

## 4）Deterministic 模块路由（按意图）

- det_process.py
  - 面向设计项目/工序范围：失效模式列表、均值指标、逐模式指标、工序内极值。

- det_extreme.py
  - 全局 S/O/D/RPN 最大最小值、RPN Top-N、阈值筛选、项目均值对比。

- det_effects.py
  - 严重度关联的失效模式/后果查询，以及后果关键词聚合。

- det_ctrl_lookup.py
  - 控制措施关键词到失效模式/项目映射，失效模式到控制项反查，控制类别判定。

- det_ctrl_presence.py
  - 预防控制与探测控制的覆盖/存在性统计。

- det_ctrl_types.py
  - 控制措施类型归纳与示例分组。

- det_causes.py
  - 失效原因语义分类、原因短语到失效模式映射、预防措施到原因反查。

## 5）设计意图总结

- 客观可查问题优先走确定性回答，降低幻觉。
- 优先图查询，向量检索作为兜底。
- API 层保持轻量，复杂逻辑下沉到 service mixin。
- 兼容旧版图结构，保持向后兼容。
- 评测脚本与在线服务逻辑解耦，便于独立回归验证。
