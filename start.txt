进程启动时会创建全局单例 rag_service = KGRAGService()，它会一直常驻内存（包含 Neo4j 连接、向量检索能力、top_k、prompt 上下文等）

3) API 层拿到 question 后调用核心逻辑

API handler 取出 question，然后调用 rag_service.answer_question(question)

4) 核心问答：先做前置检查

空问题直接拒绝（400）

检查逻辑在 kg_rag.py:356-364，问答入口在 kg_rag.py:757-777
5) 检索阶段：优先“让模型写 Cypher 去查图”，不行再用向量检索兜底

(a) 让 LLM 生成 Cypher

把你的问题塞进“生成 Cypher 的 prompt”上下文 context_cypher（里面还带了图 schema）
见 kg_rag.py:383-400 与 kg_rag.py:781-799

见 kg_rag.py:433-494
从模型回复里提取   包起来的 Cypher，并先跑一次 validate_cypher 试执行验证
见 code/kg_rag.py、code/kg_rag.py
如果可用：执行该 Cypher 得到图查询结果，并按 top_k 截断
(b) 如果 Cypher 无效或查不到结果：走向量检索

调用 similarity_search(question, k=top_k)（Neo4jVector 在 Neo4j 里做向量近邻检索）

6) 生成阶段：先“摘要压缩检索结果”，再“用摘要回答”

对每条检索结果（无论来自图查询还是向量检索），先用 ANSWER_SUMMARIZE_TEMPLATE 做一次摘要（减少噪声、方便后续回答）
见 code/kg_rag.py、code/kg_rag.py
把“问题 + 多条摘要”塞进最终 QA prompt（CYPHER_QA_TEMPLATE），再调用一次 run_inference 生成最终自然语言回答（temperature=1.0）
见 code/kg_rag.py、code/kg_rag.py
返回给前端/调用方：answer（最终答案）+ context（摘要）+ context_raw（原始检索结果）
见 code/kg_rag.py
补充：你之前“建图”做了什么，为什么后面能向量检索

建图接口把 CSV 每行 MERGE 成 5 类节点 + 4 类关系，然后为每个 FailureMode 生成一段“chunk 文本”，写入 Neo4j 的向量索引，并把 FailureMode 与 Chunk 用 :isIndexed 关系连起来，供 similarity_search 使用
见 code/kg_rag.py

训练是谁做的？

大模型（比如你这里默认 OPENAI_MODEL=gpt-4o-mini）是在 OpenAI（或你配置的 OPENAI_API_BASE/OPENAI_BASE_URL 服务方）那边提前训练好的。
你每次调用只是“推理/生成”（inference），输入 prompt，模型输出回答。

你的数据起什么作用？

你的 FMEA CSV 数据被你“建图”写进 Neo4j，并建立向量索引。
提问时程序先去 Neo4j 检索相关内容（图查询或向量相似检索），把检索到的内容拼进 prompt，发给大模型，让它“基于检索到的上下文”生成答案。
这叫 RAG（检索增强生成）：用数据做“外部知识库”，不是用数据去训练/微调模型。
什么时候才算“训练/微调”？

如果你要让模型长期学会某种风格/规则/领域知识，一般需要**微调（fine-tuning）**或其它训练方式（这套代码目前没做）。
你现在做的是“把最新数据喂给模型作为上下文”，所以数据更新后无需训练，只要重建图/索引即可。



这套系统的工作流程可以概括为以下几个核心步骤：
提问：用户提出一个关于FMEA的专业问题（例如，“某型号电池的潜在失效模式有哪些？”）。
检索：系统不会直接把整个知识图谱扔给大模型。相反，它先用用户的问题去检索知识图谱（特别是其向量嵌入库），找到与问题最相关的几段信息（即“知识片段”或“上下文”）。
增强：系统将检索到的相关FMEA知识和用户的原始问题组合在一起，形成一个新的、更丰富的提示词。
生成：这个组合好的提示词被发送给通用的、未经FMEA专门训练的大模型（如GPT-4）。大模型基于这个“被知识增强过的”提示词来生成回答。
