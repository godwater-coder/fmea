当前系统问题与后续改进方向

一、当前直接问题

1. 需要对输入的 CSV 做预处理
当前导入流程对原始表格格式依赖较强，不同来源的数据容易出现列名不统一、空值、尾部空格、编码差异等问题，从而导致建图错误、评分偏差或查询异常。

2. 前端界面仍需优化
当前系统更偏向开发验证状态，若要用于应急场景，前端还需要补足更清晰的查询入口、证据展示、结果分级和操作引导。

二、通用 KGRAG 应急系统的关键缺口

1. 缺少领域无关的查询中间表示
当前用户问题基本直接进入 Cypher 生成或向量检索，查询链路过于贴近底层执行层，不利于跨领域复用与稳定扩展。
更稳妥的做法是先将自然语言问题转换为统一查询意图，再进入检索与执行阶段。
建议抽象的中间表示至少包含以下要素：
- 实体
- 指标
- 约束条件
- 时间/作用范围
- 期望输出类型
# finished #

2. 缺少领域无关的证据裁决框架
不同领域的知识内容可以不同，但证据裁决逻辑应该统一，否则系统一旦跨领域，答案质量会明显失控。
建议建立统一的证据优先级和裁决规则，例如：
- 结构化字段命中优先
- 图关系命中次之
- 向量文本只作为候选证据
- 证据冲突时拒答
- 证据不足时返回待确认项，而不是强行生成结论

3. 缺少答案可信度分级
当前系统更偏向“尽量回答”，但应急场景下不是所有问题都应该直接输出最终结论。
建议将答案至少划分为以下三个层级：
- 可直接确认
- 高概率但需确认
- 无法确认

4. 缺少面向应急场景的固定输出协议
系统越想支持多领域、多生产场景，就越不能只依赖自由文本回答。
建议统一输出结构，至少包括：
- 结论
- 命中证据
- 适用范围
- 不确定点
- 建议下一步

5. 缺少 schema 约束和执行约束
当前 LLM 在生成查询时仍可能编造不存在的标签、属性和关系类型。
这一问题在多领域扩展后会更严重，因为 schema 差异会进一步放大模型幻觉风险。
后续应改为让模型“从已有 schema 中选择元素”，而不是自由发明 schema 元素。

三、标准 FMEA 范式方向

1. 不能让每个领域直接定义独立 schema
如果每进入一个新领域就重新定义一套图结构、字段和查询方式，系统后续会迅速失控，模型约束也难以统一维护。

2. 应先建立统一的 FMEA 核心范式
更合理的做法是先依据标准 FMEA 分析表，整理出一个跨领域通用的核心范式。
之后每次导入新的 FMEA 表格时，先做标准化、归一化和字段映射，再统一落到这一套标准格式中。

3. 系统应只面向统一后的标准 schema 工作
这样模型只需要做一次核心 schema 约束，后续不同领域只要先映射到统一范式，就可以复用同一套查询与回答框架。

4. 统一范式不能过于僵硬
实际落地时不应只有一个死板表格，而应设计为：
- 核心 FMEA 公共字段
- 受控的领域扩展字段
这样既能保证跨领域共用，又不会丢掉行业特有信息。

四、多领域数据隔离策略

第一阶段：实例级隔离
当前阶段可以采用“一个测试数据集对应一个独立 Neo4j 实例”的方式。
目标是优先保证不同领域、不同项目、不同测试集之间物理隔离，避免图查询、向量检索和统计结果相互污染。
这一阶段适合实验验证、功能开发和效果对比，优点是简单直接、排错容易，但运维成本较高，不适合长期扩张。

第二阶段：数据库级隔离
后续可升级为“同一 Neo4j 服务下，不同数据集对应不同 database”。
例如按领域或项目拆分为多个独立数据库，在保持数据隔离的同时，减少多实例带来的部署、端口、账号、资源管理复杂度。
这一阶段适合从实验系统过渡到可维护的工程化系统。

第三阶段：作用域级隔离
最终目标应是“统一服务 + 强作用域约束”。
即在同一数据库或统一服务中，通过 domain、dataset_id、project_id、version 等字段为每条知识显式打标签。
所有查询、检索、统计和生成流程在执行前都必须绑定作用域，确保系统只在当前领域、当前数据集、当前版本内工作。
这一阶段才是面向多领域通用系统的长期架构目标。
